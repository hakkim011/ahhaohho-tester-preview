## 1. 개발자 작업 — User Property 태깅

체험단 어드민에서 참여자 등록 / 탈락 처리 시 서버가 Amplitude HTTP Identify API를 자동 호출하여 아래 두 가지 user property를 설정.

**설정할 User Properties**

| Property | 타입 | 값 | 설명 |
|----------|------|-----|------|
| `is_tester` | Boolean | `true` / `false` | 체험단 여부 |
| `tester_cohort` | Number | `1`, `2`, `3`… | 기수 구분 |

**API 호출 예시 (Server-side / HTTP)**

```
POST https://api2.amplitude.com/identify
Content-Type: application/x-www-form-urlencoded

api_key=<API_KEY>&identification=[{
  "user_id": "<user_id>",
  "user_properties": {
    "$set": {
      "is_tester": true,
      "tester_cohort": 1
    }
  }
}]
```

**설정 시점**

- 어드민에서 참여자 등록 시 → `is_tester: true`, `tester_cohort: n` 설정
- 어드민에서 탈락 처리 시 → `is_tester: false` 업데이트
- 기수 종료 후에도 `is_tester: true` 유지 (배지 영구 보존 정책) — 탈락과 구분
- 탈락 시 `tester_cohort` 유지 — 기수 기록 보존 목적
- 중도 참가(기수 도중 합류) 시 → 등록 즉시 `is_tester: true`, `tester_cohort: n` 설정 (동일 기수 기준 적용)

---

## 2. Amplitude 코호트 생성 — 운영자 작업

태깅 완료 후 Amplitude에서 코호트 생성.

**전체 체험단 코호트**

```
Audiences → Cohorts → Create Cohort
조건: User Property / is_tester / = / true
```

**기수별 코호트**

```
조건: User Property / is_tester / = / true
  AND User Property / tester_cohort / = / 1
```

---

## 3. 활용

- **Spot check**: Amplitude 분석 필터에 코호트 적용 → 체험단 계정 활동만 추려서 확인
- **기수별 비교**: `tester_cohort` 값으로 1기/2기/3기 행동 패턴 비교 가능
- **전환율 추적**: 체험단 종료 후 일반 유저로 전환 여부 모니터링
